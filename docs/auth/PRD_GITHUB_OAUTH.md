# Hexo CMS GitHub OAuth 统一认证 PRD

> 版本: v1.0  
> 状态: Draft  
> 日期: 2026-05-09  
> 适用范围: Web + Desktop

---

## 1. 背景与问题

Hexo CMS 当前在认证上存在双轨模型:

1. Web 端: Better Auth + GitHub OAuth
2. Desktop 端: 本地 token/PAT 存储 + 手动输入

该模型带来如下问题:

1. 用户体验不一致: 两端登录流程完全不同。
2. 心智负担高: 用户需要理解 OAuth 和 PAT 两套机制。
3. 产品语义冲突: 文档强调 OAuth 登录，但桌面端主流程仍依赖手动 token。
4. 维护成本高: 认证逻辑分散在多层，易出现状态不一致。

本 PRD 目标是统一认证产品定义: Web 与 Desktop 都必须通过 GitHub OAuth 完成登录与授权。

---

## 2. 目标与非目标

### 2.1 目标

1. 统一认证入口: Web/Desktop 均使用 GitHub OAuth 登录。
2. 统一认证语义: 登录状态均表示“已完成 GitHub OAuth 授权并可访问目标仓库”。
3. 移除手动 token 主路径: 不再把 PAT 作为默认或推荐登录方式。
4. 降低失败率: OAuth 失败场景有清晰提示和可恢复路径。
5. 保证安全边界: Access token 仅在受控环境保存和使用。

### 2.2 非目标

1. 本阶段不引入多身份提供商 (如 Google、GitLab)。
2. 本阶段不覆盖企业 SSO、组织级 RBAC。
3. 本阶段不定义插件系统认证扩展协议。
4. 本文档不展开具体代码实现细节。

---

## 3. 目标用户与核心场景

### 3.1 用户画像

1. 个人博主: 使用单个 GitHub 仓库管理 Hexo 博客。
2. 技术内容创作者: 需要在 Web 与 Desktop 间无缝切换。
3. 开发者用户: 对 OAuth 安全性和流程可追踪性有要求。

### 3.2 核心场景

1. 首次使用: 用户首次进入应用并完成 GitHub OAuth 登录。
2. 跨端使用: 用户在 Web 登录后，在 Desktop 使用同一账号继续管理。
3. 权限不足: 用户授权范围不满足 repo 操作需求时可重新授权。
4. 会话失效: token 过期或撤销后，用户可快速恢复到可用状态。

---

## 4. 产品原则

1. OAuth First: 所有登录能力默认走 OAuth。
2. One Identity Model: 登录态只表达一种身份语义，不混用 PAT 身份。
3. Progressive Disclosure: 高级问题可见，但默认路径尽量短。
4. Recoverable Failure: 所有失败必须可恢复，不让用户卡死。
5. Platform Parity: Web/Desktop 在关键路径行为一致。

---

## 5. 需求范围

### 5.1 必须支持 (Must)

1. 登录页统一提供“使用 GitHub 登录”主按钮。
2. Web 端支持 OAuth 跳转、回调、会话建立、登录态恢复。
3. Desktop 端首版采用 GitHub Device Flow 完成 OAuth 授权、会话建立、登录态恢复。
4. 登录后必须能够执行 GitHub 仓库读写操作 (在授权范围内)。
5. 登录失败必须展示可理解错误信息与重试入口。
6. 支持退出登录，且退出后无法继续执行受保护操作。
7. 支持权限变更后的重新授权。

### 5.2 应该支持 (Should)

1. 登录成功后自动回到用户之前目标页面。
2. 在设置页展示当前授权状态与账号基础信息。
3. 在 token 失效时自动引导用户重新登录。

### 5.3 可以支持 (Could)

1. 在首次授权后展示简短“下一步配置仓库”引导。
2. 提供登录事件基础埋点 (成功率、失败原因分布)。

---

## 6. 明确废弃项

以下能力不再作为主路径保留:

1. Onboarding 中手动输入 GitHub PAT 作为登录前提。
2. Settings 中“保存 Token/删除 Token”的主要交互。
3. 将“是否有本地 token”直接等同于“已登录”。

说明:
不保留 PAT 作为应用内登录或开发模式入口。调试场景如需临时 token，应通过独立脚本或测试 mock 完成，不进入产品 UI 和主应用认证流程。

---

## 7. 用户流程 (产品视角)

### 7.1 Web 首次登录流程

1. 用户进入登录页。
2. 点击“使用 GitHub 登录”。
3. 跳转 GitHub 授权页并授权。
4. 回调成功后进入应用。
5. 若无仓库配置，引导进入仓库配置步骤。
6. 完成后进入仪表板。

### 7.2 Desktop 首次登录流程

1. 用户进入登录页。
2. 点击“使用 GitHub 登录”。
3. 应用展示 GitHub 设备授权码、授权入口和等待状态。
4. 应用打开系统浏览器，引导用户在 GitHub 完成设备授权。
5. 用户授权成功后，Desktop 自动感知授权完成并建立本地会话。
6. 若无仓库配置，引导进入仓库配置步骤。
7. 完成后进入仪表板。

### 7.3 会话恢复流程

1. 用户再次打开应用 (Web 或 Desktop)。
2. 系统检查会话有效性。
3. 有效则直接进入上次工作上下文。
4. 无效则跳转登录页并提示原因。

### 7.4 重新授权流程

1. 系统检测到权限不足或凭证无效。
2. 显示“重新授权”提示。
3. 用户重新走 GitHub OAuth。
4. 成功后回到失败前操作。

---

## 8. 状态模型

统一定义以下认证状态:

1. `anonymous`: 未登录。
2. `authenticating`: OAuth 流程进行中。
3. `authenticated`: 已完成 OAuth，会话可用。
4. `reauthorization_required`: 会话存在但权限不足或凭证失效。
5. `error`: 登录异常，需用户干预。

状态转换要求:

1. 任何受保护 API 只允许 `authenticated` 访问。
2. `reauthorization_required` 必须有明确的 CTA。
3. `error` 态不得吞错，必须有可读文案和重试入口。

---

## 9. 交互与文案要求

### 9.1 登录页

1. 主按钮文案统一为“使用 GitHub 登录”。
2. 禁止展示“认证服务未配置”这类内部实现泄漏文案给普通用户。
3. Desktop 登录中应展示设备授权码、授权入口、等待状态、取消入口。
4. 失败提示应为用户可执行文案，例如:
   - “GitHub 授权失败，请重试”
   - “授权已过期，请重新登录”
   - “当前授权缺少仓库权限，请重新授权”

### 9.2 设置页

1. 展示当前登录身份和授权状态。
2. 提供“重新授权”“退出登录”。
3. 不展示 PAT 输入框作为常规能力。

### 9.3 Onboarding

1. Onboarding 聚焦仓库配置，不要求用户手动粘贴 token。
2. 若未登录，进入 Onboarding 前应先完成 OAuth。

---

## 10. 安全与合规要求

1. Access token 不得以明文持久化在前端可读存储中。
2. 认证相关日志不得输出 token 明文。
3. 退出登录必须失效会话并清理本地可用凭证。
4. 授权范围最小化，至少覆盖博客所需 repo 操作。
5. 对 token 失效、撤销、权限变化有可观测的错误分类。

---

## 11. 验收标准 (UAT)

### 11.1 功能验收

1. Web 端用户可仅通过 OAuth 登录并完成完整内容管理流程。
2. Desktop 端用户可仅通过 GitHub Device Flow 登录并完成完整内容管理流程。
3. 用户无需手动输入 PAT 也能完成核心闭环。
4. 退出后访问受保护页面会跳回登录页。
5. 权限不足时系统引导重新授权并可恢复。

### 11.2 一致性验收

1. Web/Desktop 登录入口与核心文案一致。
2. 认证状态语义一致，不出现“一端已登录、一端仅有本地 token”这类冲突定义。
3. 文档口径一致: README、项目规划、架构文档的认证描述不冲突。

### 11.3 安全验收

1. 无 token 明文泄漏到 UI 日志、错误栈、网络响应体。
2. 未登录状态无法通过绕过 UI 访问受保护数据操作。

---

## 12. 指标 (发布后追踪)

1. 登录成功率 (Web/Desktop 分端统计)。
2. Web OAuth 回调失败率。
3. Desktop Device Flow 授权超时率。
4. 首次登录到可编辑状态的中位耗时。
5. 因权限不足触发重新授权的比率。
6. 认证相关支持请求数量 (周维度)。

---

## 13. 风险与缓解

1. 风险: Desktop Device Flow 比浏览器回调多一步输入授权码，首次体验略长。
   - 缓解: 自动打开系统浏览器，清晰展示授权码、等待状态和重试入口。

2. 风险: 历史用户依赖 PAT 流程，迁移期可能不适应。
   - 缓解: 提供迁移提示和短期兼容窗口 (仅开发模式)。

3. 风险: 文档与实现不同步导致认知混乱。
   - 缓解: 以本 PRD 为唯一产品口径源，并在实施后统一回写各文档。

---

## 14. 产品决策与开放问题

### 14.1 已确认决策

1. Desktop 首版 OAuth 采用 GitHub Device Flow。
2. Desktop 首版不采用自定义协议回调或 localhost 回调作为主路径。
3. Web 与 Desktop 仅共享 GitHub OAuth 身份语义，不共享同一套会话基础设施。
4. Web 使用服务端会话；Desktop 使用本地会话，并保留零部署、独立运行能力。
5. 不保留 PAT 作为应用内登录或开发模式入口；调试场景如需临时 token，应通过独立脚本或测试 mock 完成，不进入产品 UI 和主应用认证流程。
6. 多账号切换不纳入本期范围；首版仅支持单 GitHub 账号登录，切换账号通过退出登录后重新登录完成。

### 14.2 待确认问题

无。
